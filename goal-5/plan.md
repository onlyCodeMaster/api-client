# Goal 5 Plan

## 需求分析

用户请求“开发任务表”。结合当前项目上下文，这不是再做一轮样式优化，而是要把项目后续开发工作按优先级整理成可执行、可排期、可验证的任务清单。

本轮目标：

- 盘点当前项目已经完成、部分完成和明显缺失的能力。
- 输出一版按 `P0 / P1 / P2` 划分的开发任务表。
- 让任务表和当前代码状态一致，不重复已经完成的工作。
- 给出每项任务涉及层、主要内容、完成标准和建议顺序。

## 当前上下文

当前项目不是空壳，已经有部分底层能力和若干可用性修复：

- 已完成的阶段性任务：
  - `P0-3 Explorer 搜索过滤`
  - `P0-4 请求编辑器基础闭环`
  - `P0-5 History 基础回放`
- 正在推进：
  - `P0-6 首次启动与空状态`
- 底层已有一定实现：
  - HTTP 请求发送
  - Bearer Auth
  - 环境变量解析
  - Cookie / Proxy / TLS
  - cURL 导入导出
  - Postman Collection 导入
  - 文件上传下载基础能力

因此本轮任务表必须反映“哪些已经做过，哪些仍是缺口”，而不是从零假设项目状态。

## 风险

- 如果不核对当前代码状态，任务表会把已完成事项重复列为待做，造成排期失真。
- 如果只列功能名，不写完成标准，后续执行会继续陷入“看起来像做了，但实际上没闭环”。
- 如果不区分前端、Bridge、Rust、数据层，后续拆分 task 会不清晰，增加返工风险。

## 执行方案

1. 创建 `goal-5/input.md`、`goal-5/plan.md`、`goal-5/tasks.md`，完整保存用户原始输入并建立本轮规划。
2. 读取当前 `goal` 状态和已完成任务，整理项目现状基线。
3. 产出按 `P0 / P1 / P2` 划分的开发任务表，包含：
   - 任务 ID
   - 任务名称
   - 涉及层
   - 主要内容
   - 完成标准
   - 预估体量
4. 做一次全面一致性检查，确认任务表没有与当前实现状态冲突，并给出建议的首个迭代顺序。

## 验证方式

- `goal-5/input.md` 逐字保留用户输入。
- `goal-5/plan.md` 包含需求分析、当前上下文、风险、执行方案、验证方式、回滚方案。
- `goal-5/tasks.md` 把任务拆成可独立验证的小任务，并记录完成内容。
- 任务表内容与当前项目状态一致：
  - 不把已完成的 `P0-3 / P0-4 / P0-5` 误列为待开发。
  - 明确 `P0-6` 正在推进。
  - 待开发项有清晰的完成标准。

## 回滚方案

- 本轮只新增 `goal-5/` 文档文件，不修改业务代码。
- 所有文件创建和编辑使用 `apply_patch`。
- 如果任务表与当前状态不一致，优先修正文档而不是扩展描述。
