# Goal 1 Plan

## 目标

在当前 Tauri 2 + React + TypeScript + Rust 的跨平台 API 调试工具基础上，继续完成核心目标，把现有项目从“有布局、有部分本地能力骨架”推进到“关键主链路真实可用且已验证”的状态。

## 当前上下文

当前项目已经具备以下基础：

- Tauri 2 + React + TypeScript + Vite 项目初始化完成
- Tailwind CSS、Zustand、TanStack Query 已接入
- 主界面已经调整为接近 Postman 的工作台布局
- 前端已有请求编辑器、响应查看器、历史记录、环境变量、Collection 管理、设置页面的基础交互壳
- Rust 侧已有本地存储骨架，包含 SQLite、文件系统环境文件、Keychain Secret 的部分桥接能力
- 已存在 Tauri command，用于加载 bootstrap、保存环境、保存 secret、记录历史
- 已新增真实 HTTP 请求执行骨架，但需要重新审计其与前端、历史记录、运行验证之间是否完整一致

## 需求拆解

“继续完成核心目标”在当前上下文下，应落到以下成功标准：

1. 前端工作台不是纯展示，而是关键流程可交互
2. 请求发送链路真实可用，而不是 mock
3. 环境变量、历史记录、Secret、本地数据层至少形成第一条真实闭环
4. `npm run build` 通过
5. `cargo check --manifest-path src-tauri/Cargo.toml` 通过
6. `npm run tauri dev` 可以启动，或若受当前运行环境限制，至少要给出明确、真实、可复现的阻塞证据
7. 对现有实现做完成度审计，确认哪些需求已完成、哪些仍未完成，避免把“看起来差不多”误判为完成

## 风险

- 前端已有部分本地 mock 行为，可能与新接入的 Rust 真请求链路重复或冲突
- 历史记录可能被前端和 Rust 双写，导致重复数据
- Secret 与环境变量模板替换可能存在边界情况
- `npm run tauri dev` 依赖本机图形与端口环境，可能出现运行态问题而不仅是编译问题
- 当前已经进行过多轮 UI 调整，可能存在结构复杂、状态分散、回归风险高的问题

## 执行方案

### 第一步：完成度审计

- 逐项盘点当前实现和原始需求
- 检查关键文件、命令、构建结果和真实行为
- 找出缺失项、弱验证项和重复逻辑
- 只选择一个最高优先级任务进入实施

### 第二步：单任务推进

- 每次只处理一个任务
- 每个任务必须有明确可验证结果
- 每个任务完成前，反复进行自查与 debug 循环，直到对实现达到高置信度

### 第三步：阶段性大检查

- 每完成三个任务，进行一次大型全面检查
- 检查前端交互、Rust command、持久化、错误处理、构建、运行链路

### 第四步：最终收尾

- 对核心目标做最终完成度审计
- 从用户视角、代码质量、安全性、可维护性、验证覆盖面重新审视
- 补足所有剩余问题后再标记 goal 完成

## 验证方式

- 静态检查：`rg`、代码审阅、依赖配置审阅
- 前端构建：`npm run build`
- Rust 编译：`cargo check --manifest-path src-tauri/Cargo.toml`
- 运行验证：`npm run tauri dev`
- 如涉及 UI 变更，优先用本地运行态验证真实交互
- 如涉及持久化，检查 SQLite、文件、Keychain 调用是否形成闭环

## 回滚方案

- 每个任务保持改动边界小且独立
- 若新任务引入回归，回滚该任务涉及的文件修改
- 如问题来自前后端耦合，则优先保留已验证通过的旧链路，再分步恢复新实现
